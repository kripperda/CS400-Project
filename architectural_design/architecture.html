Proposed Architecture

The simplest set up for our application is a three-tier architecture.
In the client layer, the front-end is the web browser or mobile application used by the end user.
The middle layer will house the server which communicates between the application and the database and perform the data pull return to the end user.
The third layer is the data tier that stores and manages the data such as a NoSQL database. This tier will host the pictures, descriptions, and information of the users looking to donate or receive donations.
This architecture is especially beneficial for a large number of users for specific locations.

Design Considerations

Scalability Challenges - Anticipated upgrades in subsequent versions

Higher Costs - Additional costs for development, deployment, and maintenance for added layer

Complex Maintenance and Upgrades - Time consuming upgrades to increase scalability with adjustments across multiple layers

Increase Latency - Additional processing steps in handling requests can lead to slower response times

Separation of Concerns

Remove currency and transaction aspect and focus solely on donations
Locations limited to 50 miles of user locations
Reuse framework for each and every listings (minimum 2 photos)
Simple user interface with primary focus on messaging between users

Cross-Cutting Concerns

Record every action a verified user performs
Check user permissions before accessing resources
Handle errors that may occur
Monitor performance of servers and databases
